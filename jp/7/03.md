このチャプターでは、あの扱い辛いバイト配列`serializedGameState`を、簡単に読めてやりとりできるものにデコンストラクトしていこう。

`ZBGameMode.sol`からアクセスできるヘルパー構造体の１つに`GameState`というものがあり、これは`serializedGameState`にある全データを含んだデータ構造体である。

ではその中身を見ていこう。

## `GameState`

`GameState`はネスト化された`struct`であり現在の対戦の全状態、そして我々が変更することを望みそうな全変数を含んでいる。 こんな感じだ:

> 注: こちらの各部分が何を行っているか理解することについてはご心配なく。ほぼ見ての通りなのですが、今後のチャプターでは現実的な例で１つ１つ噛み砕いていきます。こちらではトップレベルの概観をご確認ください。

```
struct GameState {
    int64 id;
    uint8 currentPlayerIndex;
    PlayerState[] playerStates;
}

struct PlayerState {
    string id;
    CardInstance[] cardsInHand;
    CardInstance[] cardsInDeck;
    Deck deck;
    uint8 defense;
    uint8 currentGoo;
    uint8 gooVials;
    uint8 initialCardsInHandCount;
    uint8 maxCardsInPlay;
    uint8 maxCardsInHand;
    uint8 maxGooVials;
}

struct CardInstance {
    int32 instanceId;
    string mouldName;
    int32 defense;
    bool defenseInherited;
    int32 attack;
    bool attackInherited;
    int32 gooCost;
    bool gooCostInherited;
}

struct Deck {
    int64 id;
    string name;
    int64 heroId;
}
```

手短にいうと: GameStateは２人のプレイヤーを含み、各プレイヤーは一連のスタッツとカードコレクションを持っている。

後ほどこの`GameState`構造体とやりとりする方法を見ていくが、まず初めに`serializedGameState`のデータを使ってこの構造体を初期化していくぞ。

## `GameState`の初期化

まず初めにやりたいことは、新たに`GameState`をメモリで宣言することだ:

```
GameState memory gameState;
```

そのあと関数`init`を使って、`serializedGameState`の全データを読み込み、`gameState`にそのデータを追加したい:

```
gameState.init(serializedGameState);
```

> 注: `init`の背後のロジックは`ZBSerializer.sol`の中にあり、やや複雑なものです。しかしこれは背後にあるものなので、カスタムゲームモードを作るために理解しようと心配する必要はありません — あなたが超オタクでない限りは 😉
>
> 重要なことですが、Zombie Battlegroundのコアゲームロジックは`ZBSerializer`と全く同じエンコーディング/デコーディングフォーマットを使用しています。そのためTゲームの２部分が通信し、ゲーム状態全体の受け渡しができるのです。

データが`gameState`に追加されたから、`gameState`のサブプロパティを使って現在のゲーム状態にアクセスできるようになった。

例えばプレイヤー１のデッキの最初のカード名を見たければ、このようにするのだ:

```
gameState.playerStates[0].cardsInDeck[0].mouldName
```

プレイヤー２の現在のオーバーロード守備値をチェックしたいならこうだ:

```
gameState.playerStates[1].defense
```

これらは今後のチャプターで必要になった時にまたやっていくからな。
