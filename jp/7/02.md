Zombie Battlegroundカスタムゲームモードのコントラクトのシェルは、右にあるようにSolidityで書かれた他のスマートコントラクトと同じように始まる。

まず注意すべきは、我々は`ZBGameMode.sol`という名前のファイルからインポートしており、このファイルも`ZBSerializer.sol`とその他いくつかのファイルをインポートしているということだ。

> 注: これらその他のファイルの中には、非常にサイズの大きいものがあります。そのためこのレッスンではそれらを含めることはしていません。ソース完全版には<a href="https://github.com/loomnetwork/zb_game_mode" target=_blank>こちらのgithubリポジトリ</a>よりアクセスしていただけます。しかしこれらファイルのコンテンツを読んで理解しようとご心配いただく必要はありません。カスタムゲームモード作成に必要な関連知識をこれから説明していきます。

我々のコントラクト`ExampleGame`は`ZBGameMode`を継承している。だからたくさんの共通のデータ構造とヘルパーメソッドにアクセスすることができるのだ。カスタムゲーム開発者としてのお主の日々をイージーにするために、これらを作成しておいたからな。

それと我々はこれをライブラリに抽象化しておいたから、各カスタムゲームモードに同じ定型ロジックを書く必要はない。

### "フック" — カスタムゲームモードの核心

全てのカスタムゲームモードにとって最初のステップとなるのは、少なくとも１つの **_フック_** を実装することだ。

コアのゲームロジックは基本的に、コードにあればゲームのある部分においてある関数をコントラクトから探し呼び出す。

例えば、`beforeMatchStart`は対戦がスタートする前にのみ呼び出され、`afterInitialDraw`はプレイヤーが最初のカードを引いた後すぐ呼び出される。`beforeTurnStart` *(計画しているが未実装)* は各プレイヤーのターンスタート時に発動する。

これらのフックは、ゲームが続きを始める前にお主のカスタムルールを適用し、その時点でのゲーム状態を「利用」して編集することを可能にするのだ。

我々が実装する最初のフックは`beforeMatchStart`というものだ。これはその名前の示すとおり、各対戦のスタート前に適用されるフックだ。

この関数定義はこんな感じだ:

```
function beforeMatchStart(bytes serializedGameState) external {
    // ゲーム状態への変更をここで定義
}
```

この関数内で、対戦スタート時に適用したいゲームロジックへのカスタム変更の全てを定義するのだ。

これらの変更というのは、個別のカードのバランス変更とか、どのカードがこのゲームモードで許可/禁止されるかというルール、さらにオーバーロードの体力やGooボトルのスタート数といったようなものとなる。

> Zombie Battlegroundをプレイしたことがなくて、用語の意味がわからないなら、<a href="https://zombiebattleground.io/index.php/Gameplay" target=_blank>Zombie Battlegroundのゲームプレイルール</a>の簡単な紹介を読んでみましょう。
